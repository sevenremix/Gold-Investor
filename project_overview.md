## 1. 项目核心愿景与定位
本项目是一个专门针对 **黄金ETF投资** 的 **双层量化配置引擎 (Two-Layer Allocation Engine)**。
由于国内黄金（如 A股 518660 工银黄金）和国际黄金（如 美股 IAUM）存在汇率波动、跨币种折溢价（沪伦价差）以及二级市场流动性摩擦，本项目通过拉取实时宏观与微观数据（TIPS, USD/CNH, RSI, IOPV等），给出数学级别的精准仓位与切分建议。

**本系统的核心产出是两个数字：**
1. **买多少？(Position Sizing)**：由 Layer 1 计算得出，范围一般为 0.0x ~ 1.5x 标准仓位。
2. **买哪个？(Vehicle Routing)**：由 Layer 2 计算得出，给出资金在 IAUM 与 518660 之间的具体百分比分配。

## 2. 核心架构与文件解析

系统严格遵循“前后端分离、逻辑与视图解耦”的设计原则：

*   **`gold_engine.py` (后端大脑/无任何UI依赖)**
    *   纯粹的数据结构 (`dataclass`) 和数学逻辑。包含了五大类核心组件：`StrategyConfig`（参数）、`MarketData`（行情数据源）、`DerivedMetrics`（中间衍生数据）、`AllocationResult`（判定结果）以及核心运行类 `GoldAllocator`。
    *   包含双层算力模型（Layer 1 专注宏观机会成本+技术情绪，Layer 2 专注汇率边际+国内溢价+品种折价）。
    *   **开发注意**：修改任何量化逻辑应首先在这里处理，且必须确保不引用任何 `streamlit` 或前端库。

*   **`data_fetcher.py` (数据抓取管道)**
    *   通过 `yfinance`, `akshare`, 新浪财经API, FRED 接口获取实时行情。
    *   **高阶逻辑注意（518660 IOPV 问题）**：由于直接获取官方实时 IOPV 数据存在难度且滞后严重，系统创新采用了 **“动态比例追踪 (Dynamic Proportional Tracking)”** 机制：用 T-1 净值去乘以 (今日 Au9999 实时现价 / 昨日 Au9999 收盘价)，同时通过 `round(..., 4)` 对齐二级市场精度。
    *   该模块具有高鲁棒性设计（Fail-safe），任何 API 请求失败均不可导致系统崩溃卡死，而应优雅地保留现有缓存数值并返回。

*   **`app.py` (Streamlit 前端交互看板)**
    *   构建精美、科技感的 Dark Mode 界面（内置深度定制的 CSS）。
    *   左侧 Sidebar 用作调参中心；主区域分为：手动行情输入/获取、衍生指标展示、带有进度条等可视化组件的配置结果输出。
    *   包含一个日志记录功能按钮，能调用后端模块生成降级报告 `gold_data_log.md`。

*   **辅助文档 (重要背景)**
    *   `strategy_logic_explanation.md`：详细记录了为什么弃用普通 IOPV 估算、以及针对微幅与深幅折价（如 -6.94%）的有效性分类解读。
    *   `factor_breakdown_analysis.md`：看板因子的详细数学推导和金融实战含义（尤其包括宏观 TIPS 的锚定原理和阻尼器效应）。
    *   `strategy_config.json`：前端策略配置的落地持久化文件。

## 3. 已知环境或运行时问题 (Troubleshooting)

*   **Streamlit 运行异常**：近期（例如 2026-03 等测试期）通过 `streamlit run app.py` 启动应用时，会出现控制台打印出本地和网络 URL 后**立即崩溃退出**(`Stopping...`, Exit code 1) 的现象。
    *   **排查线索**：这不是代码语法级别的错误（若语法错误往往伴随着 Python Traceback）。可能原因涉及 Windows 终端/环境变量、所依附的包 `beautifulsoup4`/`akshare`/`yfinance` 导致的底层 C++ 库（如 Numpy）二进制冲突，或者是端口被意外占用。
    *   **后续接手提示**：如果您被要求排查这个问题，请不要大改 Python 业务逻辑代码，而应从环境依赖 (`pip list`，重装 streamlit，隔离虚拟环境)、端口探查等方面入手。

## 4. 后续开发指导与底线原则
1. **UI 审美底线**：`app.py` 已经构建了极具赛博/金融工程师风格的视觉 UI，请不要随意删减 CSS，后续任何增加的原生组件必须用类似的 HTML 标签包覆以注入样式。
2. **逻辑不可变幻**：不要擅自利用机器学习库（如 sklearn）代替当前的明确参数（如权重 0.6、0.4）方程。本系统主打的是基于规则的透明性算式，如需引入动态回测框架应另开分支。
3. **保持防呆 (Fail-Safe)**：金融系统忌讳单点崩溃。如果要接入新的数据源（如更换某行情接口），必须包裹 `try-except` 或保持类似 `fallback_data` 的设计。
